# 看门狗（Watchdog）

Zigbee2MQTT 支持一个简单的看门狗（watchdog），用于处理"软故障"（soft failures，即 Zigbee2MQTT 可以在不崩溃的情况下妥善处理的故障，例如"适配器已断开连接"）。正常情况下需要重启 Zigbee2MQTT 进程，或者重置容器（对于容器化安装）。这个看门狗允许 Zigbee2MQTT 在内部尝试重启，而无需退出进程。此功能为可选项，只有在设置了有效的 `env` 变量时才会启用。

```bash
Z2M_WATCHDOG=default
```

以看门狗的默认重试延迟（1分钟、5分钟、15分钟、30分钟、60分钟）启动 Zigbee2MQTT。

配置的延迟数量实际上就是看门狗重试的次数，超过这个次数后，node 进程将会停止（以避免在明显需要用户注意的情况下无休止地重试）。
看门狗会在第一次失败后 1 分钟重试，第二次失败后 5 分钟重试，第三次失败后 15 分钟重试，第四次失败后 30 分钟重试，第五次失败后 60 分钟重试，如果第六次启动仍然失败则退出。任何一次成功启动都会将计数重置到起点。

此外，以下行为始终适用：

- 只有在初始的（手动）启动成功之后，看门狗才会在失败时触发。
- 设置（settings）出现问题时将始终忽略看门狗并停止 Z2M。
- 手动停止/重启（例如 `CTRL+C`）将忽略看门狗，以遵从用户的意图。

::: tip 提示
在非容器化环境中，为了处理 NodeJS 崩溃，你需要在操作系统上使用一个专门的看门狗程序，以便自动重启 Zigbee2MQTT 进程。
:::

## 自定义延迟

```bash
Z2M_WATCHDOG=minutes_csv
```

使用你自己选择的看门狗重试延迟启动 Zigbee2MQTT。
`minutes_csv` 应为一个以逗号分隔的数字列表，表示分钟数（也可以是分钟的小数部分，使用小数点 `.`）。其他数值/格式将阻止 Z2M 启动。

示例：

```bash
Z2M_WATCHDOG=5,10,30
```

以看门狗的重试延迟 5 分钟、10 分钟、30 分钟启动 Zigbee2MQTT。这将产生 3 次重试尝试。

```bash
Z2M_WATCHDOG=0.5,3,6,15
```

以看门狗的重试延迟 30 秒、3 分钟、6 分钟、15 分钟启动 Zigbee2MQTT。这将产生 4 次重试尝试。

```bash
Z2M_WATCHDOG=1
```

以看门狗的重试延迟 1 分钟启动 Zigbee2MQTT。这将产生 1 次重试尝试。
